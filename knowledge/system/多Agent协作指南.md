## 核心架构

### 三大核心概念

| 概念 | 定义 | 类比 |
|------|------|------|
| Accounts | 飞书开放平台的独立企业自建应用 | 团队里的每位正式员工 |
| Agents | OpenClaw侧的独立AI大脑 | 员工的专业能力与大脑 |
| Bindings | 飞书账号与Agent的映射关系 | 工位与员工的对应关系 |

### 合规互@核心工作流程

```
1. Agent A通过飞书官方API，发送携带目标Agent B正确open_id的标准化@消息
2. 飞书平台校验消息合规性后，通过WebSocket长连接推送给Agent B
3. OpenClaw通过Bindings路由，将消息精准分发到Agent B的独立工作区
4. Agent B解析消息中的@标签，确认自己被触发后，执行任务
5. 全程通过A2A协议实现任务上下文同步
```

## 合规要求

### 飞书@消息格式（必须严格遵守）

```
必须使用：<at user_id="目标机器人open_id">目标机器人名称</at>
禁止使用：纯文本@、@昵称、@user_id
```

### 必须开通的权限

| 权限代码 | 权限名称 | 核心用途 |
|---------|---------|---------|
| im:message:send_as_bot | 以应用身份发送消息 | 机器人发送带@的消息 |
| im:message.group_at_msg:readonly | 接收群内@机器人的消息 | 识别@自己的消息 |
| im:message.p2p_msg:readonly | 读取用户发给机器人的单聊消息 | 私聊调试与配对 |
| im:chat:readonly | 获取群组信息 | 识别群内成员 |
| contact:user.base:readonly | 获取用户基本信息 | 解析@标签对应的机器人身份 |

### 合规红线

1. ❌ 禁止使用群内自定义机器人（Webhook机器人）
2. ❌ 禁止多个Agent共用同一个飞书应用
3. ❌ 所有权限必须先申请、再使用

## A2A协议

### 核心配置

```json
"tools": {
  "agentToAgent": {
    "enabled": true,
    "allow": ["agent1", "agent2", ...],
    "allowCrossWorkspace": true,
    "autoSyncContext": true
  }
}
```

### 作用

- 让多个Agent能够互相发现
- 共享上下文
- 管理任务生命周期
- 配合飞书官方@消息规范

## Agent协作规则

### 公开会议协作规则

1. 收到会议/任务需求时，先在群内明确会议主题与议程
2. 按照分工，依次@对应负责人，明确告知任务与回复时限
3. 必须等待上一位负责人回复完毕后，再@下一位负责人
4. 所有成员发言完毕后，汇总所有内容，形成会议结论，@用户汇报

### @消息发送模板

```
> 【任务指派】
> <at user_id="ou_xxxxxx">目标Agent名称</at> 请你针对本次需求，输出技术实现方案，重点评估开发周期与潜在风险，预计10分钟内回复。
```
