# GITHUB_GOVERNANCE.md · GitHub 仓库治理规则

> 更新时间：2026-04-27 18:19 CST | 维护：太阳

## 一句话原则
GitHub 上看到的内容，才算正式结果。

## 权限边界
仅以下两位可以直接操作 GitHub 仓库：
- 太阳（COO / 总调度）
- 光头强（系统官 / 技术执行）

除以上两位外，其他任何角色：
- 不得直接上传
- 不得直接 push
- 不得直接删除远端内容
- 不得直接修改仓库可见性
- 不得直接改主分支内容

## 协作机制
太阳与光头强共同负责 GitHub 仓库治理，并建立固定协作协议，至少覆盖：
1. 什么内容允许上传，什么内容禁止上传
2. 谁在什么情况下负责上传
3. 谁忙不过来时如何交接
4. 上传前如何做风险检查（隐私/敏感信息/误删）
5. 上传后如何验收，以 GitHub 可见结果为准

## 默认分工
- 太阳：定规则、定优先级、定保留/删除/更新策略、最终验收
- 光头强：执行上传、同步、技术排障、仓库结构修复

## 上传原则
- 本地文件不等于正式资产
- GitHub 可见结果才算正式资产
- 未经太阳与光头强确认，不上传敏感资料
- 删除、公开/私有切换、覆盖远端内容，属于高风险动作，必须谨慎

## 结果汇报统一口径
以后只用以下 3 个词汇报 GitHub 动作：
1. 升级上传 = 比以前更完整、更清楚、更成熟
2. 新增上传 = 原有不变，只增加新的
3. 有回退风险 = 会打破原来已设置好的东西，接近推翻重建

## 默认上传原则
- 默认：保留已有内容，新增优化内容
- 没有明确确认，不做破坏原有基础的重建式上传
- 本地已整理后，只问一句：现在要上传到网上，要不要同步？


## GitHub 协作铁律（新增）
- 7 个核心分身都可以阅读 GitHub 仓库中的规则制度与公开文档。
- 除太阳、光头强外，其他分身默认没有直接上传 GitHub 的权限。
- 分身如有新增或更新内容，先提交给太阳验收；由太阳统一检查、归档、上传。
- 光头强负责技术支持、同步、排障；是否上传、上传什么，由太阳最终拍板。
- 禁止分身各自为政地直接改 GitHub，避免仓库混乱、版本失控、口径不一致。
