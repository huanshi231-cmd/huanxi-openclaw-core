## 沟通与汇报规则

- **先结果，后解释**：汇报时第一句是结论/结果，解释和原因放后面
- **不问废话**：接任务自己补全意图，只有一个关键信息真的不确定才问一个问题
- **不把问题抛回给欢喜**：遇到不确定自己判断，不让欢喜替你想
- **不说可能/也许/应该**：没查过不能猜，查到了再说

## 执行铁律

- **配置修改必须先停网关**：`launchctl bootout gui/501/ai.openclaw.gateway` → 改文件 → `launchctl load ~/Library/LaunchAgents/ai.openclaw.gateway.plist`
- **禁止闭卷生成**：说任何配置/状态/文件内容，必须先调工具查证，不能凭印象
- **一次做到100%**：接任务先完整读所有要求，想清楚了再开始，不边做边补充
- **不交半成品**：交付前自检"这个东西现在能用吗"，不能用就不交

## GitHub 规则

- **改本地→commit→push才算完成**：本地修改不算，GitHub可见才算
- **保留已有内容，新增优化**：上传前检查，不覆盖，只叠加
- **敏感数据不进Git**：边界以 UPLOAD_BOUNDARY.md 为准
- **只有太阳、光头强可以直接操作GitHub**

## 错误处理规则

- **出现失败先查实证**：status/logs/config/命令输出，找到具体原因再说
- **每次犯错必须写进错误库**：当次对话结束前写入 `错误库.md`，不写是失职
- **遇到类似场景先查错误库**：接任务前先搜有没有对应的历史错误

## 记忆沉淀规则

- **每次欢喜纠正理解，当次对话必须写进 NATURAL_LANGUAGE_DECODER.md**
- **新经验、新偏好，当次对话写进对应记忆文件**

## 飞书多Bot群协作踩坑规则（2026-04-28）

### 规则1：群内@Bot无反应 → 先查路由配置
- 根本原因：OpenClaw网关没有配置 @mention 路由规则
- 正确做法：bindings 里给每个分身加群绑定，重启网关验证
- 新增Bot必须第一时间加路由，不加不上线

### 规则2：@标签 user_id 不要手动转义引号
- 根本原因：手动转义 + json.dumps 二次转义 → user_id 变空
- 正确做法：直接写原生字符串，交给 json.dumps 处理
- 发完必查飞书响应的 mentions 数组，有返回=@成功

### 规则3：open_id 与 numeric_id 严格区分
- open_id：App视角隔离，每个Bot看到的同一用户open_id不同，用于实际发消息/路由配置
- numeric user_id：全局唯一，用于文档记录
- 绝不混用

### 规则4：Bot是否在群 → 发消息测试，不要查API
- 飞书群成员API不返回Bot，查不到是正常现象
- 金标准：发一条@消息，飞书响应的mentions数组有返回=Bot在群里

## 番茄小说上传踩坑记录（2026-04-29）

### 踩坑1：CDP自动化无法触发React受控组件
- **现象**：番茄小说编辑器的章节号输入框是React受控组件，用CDP自动化设置DOM值后，React内部状态不更新，"下一步"按钮校验失败
- **已尝试**：nativeInputValueSetter + InputEvent、直接调用React fiber的onChange、模拟键盘输入，全部无效
- **结论**：React受控组件的自动化填写需要特殊处理，常规DOM操作不行
- **建议**：需要人工手动输入章节号，或研究React状态注入方案

### 踩坑2：分身不会用飞书原生@功能
- **现象**：开心果不知道飞书里直接输入@就能选人，想手动复制open_id来@
- **正确方式**：飞书输入框直接打@符号，弹出成员列表选人，不用记ID
- **系统侧**：系统消息用 `<at user_id="ou_xxx">名字</at>` 格式，但分身不需要自己写这个

### 踩坑3：卡在一个问题死磕不停
- **现象**：CDP连不上、React状态触发不了，反复尝试十几遍不暂停
- **教训**：发现反复卡住 → 先停下来，回头查规则和踩坑记录，不要死磕
- **欢喜原话**："反复卡在一个上面，你们不觉得发现有问题吗？那就先暂停一下呀"

### 踩坑4：分身分不清自己的文件在哪
- **现象**：开心果说"等蕊蕊发第4章内容"，但内容已经在 `/Users/huanxi/.hermes/fanqie/chapter4.md`
- **教训**：分身要先查自己本地文件，再等群内发送
