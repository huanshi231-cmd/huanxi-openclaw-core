## 二、4月19日重大教训

### Lesson 8：model配额耗尽是系统性风险，需要完整failover链

**问题**：volcengine和kimi-k2.5同日先后耗尽配额，导致neirong批量cron超时

**根因**：
1. 主力模型双挂 → failover到MiniMax → MiniMax高频任务也承压
2. 大任务（>60分钟）超时 → 所有model均超时
3. cron任务重试不力（未配置max retries）

**解决方案（已落地）**：
- 冷却机制：连续失败≥2次 → 6小时冷却，不再空转
- learnings检索：失败时自动查70个历史learnings
- system doctor：一键诊断cron+冷却+历史经验

**触发式规则**：
```
当：发现 model 返回 402/429/500（配额/余额问题）
执行：
  1. 立即切换到下一个可用model
  2. 触发冷却机制（≥2次失败）
  3. 并行触发 learnings 检索
  4. 若影响cron任务 >3个，立即飞书告警
不允许：连续3次调用同一配额耗尽的 model
```
