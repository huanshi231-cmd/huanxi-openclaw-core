## 1. 这个仓库是做什么的？

这个仓库是欢喜「一人公司 AI 团队」的核心配置备份库。

它保存的是：

- 每个核心机器人的身份定位

- 团队角色总表
- 头像设计任务书
- 上传边界与同步流程

一句话：

> 这里是 AI 团队的「岗位说明书 + 公司制度 + 可恢复备份」。

## 4. 每个机器人为什么有 7 个核心文件？

每个机器人目录下都有这 7 个文件：

| 文件 | 作用 |
|---|---|
| `IDENTITY.md` | 说明「我是谁、叫什么、负责什么」 |
| `SOUL.md` | 说明「我的性格、说话方式、价值观」 |
| `AGENTS.md` | 说明「我怎么接任务、怎么执行、怎么汇报」 |
| `USER.md` | 说明「服务对象欢喜是谁、偏好是什么」 |
| `TOOLS.md` | 说明「我能用什么工具、工具边界是什么」 |
| `MEMORY.md` | 说明「我必须长期记住什么」 |
| `HEARTBEAT.md` | 说明「我心跳巡检时要检查什么」 |

这 7 个文件合起来，就是一个 AI 员工的基础运行说明。

## 5. 什么可以上传？

允许上传：

- 机器人名称、花名、职责、风格
- SOP、协作规则、岗位说明
- 通用模板、非敏感流程
- 头像设计 brief
- 非敏感 Skill 说明
- 可公开复用的 H5/页面/代码模板

---

## 6. 什么绝对不能上传？

禁止上传：

- 私密对话、聊天记录
- 客户资料、个案内容、咨询记录
- KPI、收入、转化数据、内部经营敏感数据
- `.env`、API Key、Token、Cookie、SSH 私钥
- mempalace / sqlite / bin / cache / logs / daily_log 等运行数据
- 未清洗的商业计划、内部策略原稿

详细边界见：[`docs/UPLOAD_BOUNDARY.md`](./UPLOAD_BOUNDARY.md)

## 7. 欢喜可以怎么用？

欢喜可以直接在 GitHub 上查看或修改 MD 文件。

例如：

- 想改灵夕的性格：改 `agents/lingxi-inspiration/SOUL.md`
- 想改跳跳的职责：改 `agents/xinmeiyunying-tiaotiao-ops/IDENTITY.md`
- 想改团队分工：改 `docs/TEAM_ROLES.md`

修改后告诉太阳：

> 太阳，同步 GitHub 最新配置。

太阳会负责：

1. 拉取 GitHub 最新版本
2. 检查有没有敏感信息
3. 检查格式是否正确
4. 同步到本地机器人工作区
5. 必要时提交一版新的备份

## 8. 当前安全策略

当前采用「人工确认同步」模式，不做完全自动同步。

原因：

- 防止错误配置自动进入运行环境
- 防止私密信息误上传/误同步
- 防止机器人文件被外部修改后直接污染本地

推荐流程：

> GitHub 可编辑，太阳审核后同步落地。

---

## 9. 备份是否等于运行？

不是。

GitHub 仓库是「云端标准版」。
本地 OpenClaw 工作区是「实际运行版」。

要让 GitHub 修改真正生效，必须同步到本地工作区。

运行关系：

```text
GitHub 仓库（标准备份）
        ↓ 太阳审核同步
本地 OpenClaw 工作区（实际运行）
        ↓
7 个机器人按新配置工作
```

---

## 10. 这一步的核心意义

这个仓库不是摆设，也不是纸上谈兵。

它的作用是：

1. 配置不丢
2. 改错可回滚
3. 团队职责可查
4. 新机器人可复制
5. 系统故障可恢复
6. 欢喜可以直接参与调整 AI 团队
7. 一人公司逐步从「临时聊天」变成「可运营系统」
