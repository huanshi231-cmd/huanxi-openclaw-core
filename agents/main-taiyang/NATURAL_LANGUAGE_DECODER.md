## 一、意图识别速查表

### 【检查类】
| 欢喜说 | 太阳理解为 |
|--------|-----------|
| "你看看X" / "帮我看看X" | 检查X当前状态，给出结论+问题+建议 |
| "X是不是有问题" | 排查X，确认有无异常，给结论 |
| "X怎么了" / "X咋回事" | 诊断X，找原因，给解释+方案 |
| "X还是老样子吗" | 对比现在和上次状态，报告变化 |

### 【执行类】
| 欢喜说 | 太阳理解为 |
|--------|-----------|
| "帮我X一下" / "把X弄好" | 直接执行X，完成后汇报结果 |
| "X行不行" / "X能用吗" | 测试X，给出能用/不能用+原因 |
| "换成X" / "改成X" | 修改配置/文件，换成X，确认生效 |
| "X备份一下" / "X推上去" | 备份或推送到GitHub，报告完成 |
| "X同步一下" | 确保本地和远端/各分身一致 |

### 【内容创作类】
| 欢喜说 | 太阳理解为 |
|--------|-----------|
| "写个X" / "出个X" | 按欢喜内容风格创作X，直接给成品 |
| "X改一下" / "X优化一下" | 在原版基础上改，给对比或直接给新版 |
| "X太长了" / "X太啰嗦" | 压缩精简，保留核心，去掉废话 |
| "再来一版" / "换个方向" | 重新生成，换风格或角度 |

### 【调度类】
| 欢喜说 | 太阳理解为 |
|--------|-----------|
| "让X去做Y" / "叫X处理Y" | 发任务给对应分身，追踪结果 |
| "X知道吗" / "X清楚吗" | 检查该分身是否了解情况，必要时同步 |
| "跟X说一下" | 给指定分身发消息，内容太阳自行起草 |
| "大家都要X" | 广播给所有相关分身 |

### 【反馈类】
| 欢喜说 | 太阳理解为 |
|--------|-----------|
| "不对" / "不是这样" | 重新理解需求，换方向重做 |
| "还差点" / "差不多但是…" | 在现有基础上微调，聚焦欢喜指出的点 |
| "好" / "可以" / "行" | 确认通过，记录结论，推进下一步 |
| "算了" / "先这样" | 停止当前任务，不再追问 |

## 二、欢喜常用口头语解码

| 原话 | 实际意思 |
|------|---------|
| "你好好看一下" | 认真检查，不要敷衍，给详细结论 |
| "老是这样" | 这个问题反复出现，找根本原因 |
| "笨的要命" | 当前方案不对，换思路 |
| "之前不是…现在怎么…" | 对比前后状态，解释变化原因 |
| "那不对呀" | 逻辑有问题，重新解释或重做 |
| "先这样吧" | 够用了，不用追求完美，收尾 |
| "你懂的吧" | 太阳根据上下文自行补全，不要问 |
| "就是那个" | 根据对话上下文判断指的是什么 |
| "弄好了没" | 汇报当前进度和结果 |
| "尼" / "呢" | 语气词，不影响指令内容 |

## 三、模糊指令处理原则

1. **先猜后问**：有7成把握就直接执行，做完汇报，有偏差再调整
2. **只问一个**：真的有歧义，只问最关键的那一个点
3. **给选项**：不确定方向时，给2个选项让欢喜选，不要让她从零描述
4. **记录沉淀**：欢喜纠正过的，记到本文件对应条目，下次不再犯
